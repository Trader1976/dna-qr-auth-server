DNA QR Authentication Protocol

Clean spec for v3 (stateful) and v4 (stateless/CDN-scale)
v3 & v4 — Stable

Device-mediated, passwordless authentication using DNA identity

This document specifies the DNA QR Authentication protocol used by a verifier (server) and DNA-Messenger (authenticator). It supports v3 (stateful session storage) and v4 (stateless verification).

1. Overview

DNA QR Authentication is a passwordless, device-mediated protocol where DNA-Messenger approves an authentication request initiated by a website or application.

Key properties

  • No passwords
  • No shared secrets
  • No cookies required
  • User approval happens on a separate trusted device
  • Identity bound to a fingerprint derived from the public key
  • Supports post-quantum signatures (ML-DSA-87)
  • Supports stateless / CDN-scale verification (v4)

Modes

Mode Name Server State Use case
v3 Stateful Required Simple deployments, legacy compatibility
v4 Stateless Not required CDN-scale, zero-storage verification

2. Actors

Actor Description
Client (Browser / App) Displays QR code and waits for approval
Authenticator (DNA-Messenger) Holds DNA identity and signs approval
Verifier (Server) Verifies signatures and grants access

3. Identity Model

Each DNA identity includes a signing keypair and a stable fingerprint derived from the public key: 

fingerprint = hex(SHA3-512(public_key))

The verifier never stores private keys. The fingerprint is the stable user identifier.

4. QR Payloads

QR codes always encode a dna:// URI.

!
Important: QR payloads never include secrets. They encode descriptors, tokens, and display hints.

4.1 Common properties

  • Transport is URI text (QR is just a carrier)
  • All validation is done by signatures and bindings, not by QR secrecy

5. Protocol v3 — Stateful Mode

5.1 Characteristics

  • Server stores session state
  • Browser polls server for approval
  • Simple deployment model
  • Requires server memory/database

5.2 QR payload (v3)

dna://auth
  ?v=3
  &origin=https://example.com
  &session_id=...
  &nonce=...
  &expires_at=...
  &rp_id=example.com
  &rp_id_hash=base64(sha256(rp_id))
  &callback=https://example.com/api/v1/session/{id}/complete

5.3 Approval flow

  1. Server creates session
  2. Browser shows QR
  3. Phone scans QR
  4. Phone signs canonical payload
  5. Phone POSTs signed payload to callback
  6. Server verifies bindings and signature
  7. Server marks session approved
  8. Browser polling redirects to success

6. Protocol v4 — Stateless Mode (Recommended)

6.1 Characteristics

  • No server session storage required for verification
  • Verifier nodes can be stateless and horizontally scaled
  • Approval token (at) is portable
  • Server needs a signing key for minting tokens

6.2 Server-signed session token (st)

The server mints a session token st, signed with Ed25519. The client and phone treat it as opaque.

st claims

{
  "v": 4,
  "typ": "st",
  "sid": "session-id",
  "origin": "https://example.com",
  "rp_id_hash": "base64(sha256(rp_id))",
  "nonce": "...",
  "issued_at": 1700000000,
  "expires_at": 1700000300
}

6.3 QR payload (v4)

dna://auth
  ?v=4
  &st=<server-signed-token>
  &origin=https://example.com     (UI hint)
  &app=ExampleApp                (UI hint)

origin and app are display hints only and not security-critical.

6.4 Phone-signed payload (v4)

The phone signs a canonical JSON object that binds approval to st using st_hash = base64(sha256(st)).

Canonical fields (exact order)

{
  "expires_at": 1700000300,
  "issued_at": 1700000000,
  "nonce": "...",
  "origin": "https://example.com",
  "rp_id_hash": "...",
  "session_id": "...",
  "sid": "...",
  "st_hash": "base64(sha256(st))"
}

Canonicalization rules

  • UTF-8 bytes
  • No whitespace
  • Exact key order
  • Standard Base64 (with padding)

6.5 Verification flow (v4)

  1. Browser requests /api/v4/session
  2. Server returns { st, sid }
  3. Browser displays QR
  4. Phone scans QR
  5. Phone signs canonical payload
  6. Phone POSTs to /api/v4/verify
  7. Server verifies st, bindings, fingerprint, allowlist (optional), and signature
  8. Server issues at approval token
  9. Browser polls status endpoint and redirects on approval

7. Approval Token (at)

The at token is server-signed and portable. It can be used to continue an auth flow or authorize API calls. 

{
  "v": 4,
  "typ": "at",
  "sid": "...",
  "st_hash": "...",
  "rp_id_hash": "...",
  "fingerprint": "...",
  "issued_at": 1700000000,
  "expires_at": 1700000060
}

8. Security Properties

8.1 Guarantees

  • Device-mediated consent
  • Origin binding
  • RP binding (WebAuthn-style)
  • Replay protection
  • Post-quantum signature security
  • No shared secrets or password database

8.2 Threat model

Threat Mitigation
QR reuse Nonce + expiry
MITM Origin + RP binding
Replay st_hash + nonce
Server compromise No user secrets stored
Quantum adversary ML-DSA-87

9. Configuration

Select protocol mode using environment variable: 

AUTH_MODE=v3 | v4 | auto

Behavior

Value Result
v3 Stateful only
v4 Stateless only
auto Prefer v4, allow v3 fallback

10. Design Goals

  • Minimal trust surface
  • Explicit cryptographic bindings
  • Human-verifiable consent
  • Server simplicity
  • Long-term cryptographic resilience

11. Summary

DNA QR Authentication is a clean alternative to passwords and traditional platform-bound authenticators, enabling device-mediated login with post-quantum signatures and optional stateless verification.

✓
Recommended: Use v4 for stateless/CDN-scale deployments. Use v3 for simplest stateful flows or legacy compatibility.
